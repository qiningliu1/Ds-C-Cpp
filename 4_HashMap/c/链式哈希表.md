# 链式哈希表（拉链寻址法）- Cornell笔记法

## 📚 主动回忆区域 | 详细笔记区域

### 基本概念理解
| 回忆提示 | 详细解答 |
|---------|----------|
| 什么是拉链寻址法？ | 每个桶是一个链表，处理哈希冲突时将相同哈希值的元素串联在同一个桶的链表中 |
| `buckets[index]`的双重身份？ | 1. 作为指针数组的第index个元素<br>2. 作为该桶链表的头节点指针 |
| 为什么用`Node**`而不是`Node*`？ | `Node**`是指向指针数组的指针，支持动态扩容；`Node*`只能指向单个节点 |

### 内存管理核心问题
| 回忆提示 | 详细解答 |
|---------|----------|
| `free(bucket->pair->key)`需要吗？ | ❌ 不需要！`key`是int类型，直接存储在结构体中，非动态分配 |
| `put`函数更新value时的内存泄漏？ | ⚠️ 必须先`free(current->pair->value)`释放旧内存，再malloc新内存 |
| 扩容时新桶为什么要初始化为NULL？ | `malloc`分配的内存内容未定义（垃圾值），不初始化会导致访问无效指针 |
| `free(bucket)`后的野指针问题？ | 虽然会产生野指针，但代码中立即用`bucket = next`覆盖，所以安全 |

### 关键操作实现
| 回忆提示 | 详细解答 |
|---------|----------|
| 删除头节点vs中间节点的区别？ | 头节点：`buckets[index] = current->next`<br>中间节点：`prev->next = current->next` |
| 为什么删除需要`prev`指针？ | 删除中间节点时需要修改前一个节点的next指针，指向被删除节点的下一个节点 |
| 扩容时为什么要重新put？ | 容量变化导致哈希值变化，需要重新计算每个元素的新位置 |

### 扩容机制深度理解
| 回忆提示 | 详细解答 |
|---------|----------|
| `oldBuckets`保存地址会被覆盖吗？ | ❌ 不会！指针变量独立存储，`map->buckets = malloc(...)`不影响`oldBuckets`的值 |
| 为什么扩容时`size`要重置为0？ | 因为要重新调用`put`函数，`put`会递增`size`，不重置会导致`size`翻倍 |
| 扩容中的内存释放层次？ | 从内到外：`value字符串` → `pair结构体` → `node节点` → `buckets数组` |

## 💡 核心总结

### 关键原理
1. **拉链寻址法**：每个桶是独立链表，通过链表处理哈希冲突
2. **动态扩容**：负载因子超过阈值时，容量翻倍并重新分布所有元素
3. **内存管理**：严格遵循"谁分配谁释放"，防止内存泄漏

### 最佳实践
1. **指针操作**：删除节点时必须维护前驱指针处理链接关系
2. **内存安全**：malloc分配的内存必须初始化，更新时先释放旧内存
3. **边界处理**：头节点和中间节点的操作逻辑不同，需要分别处理

### 常见陷阱
1. ❌ 忘记释放字符串内存导致泄漏
2. ❌ 删除中间节点时链接关系错误
3. ❌ 扩容时忘记初始化新桶为NULL
4. ❌ 误认为int类型需要free

### 时间复杂度
- **平均情况**：O(1) - 插入、删除、查找
- **最坏情况**：O(n) - 所有元素哈希到同一个桶
- **扩容操作**：O(n) - 需要重新分布所有元素

### 空间复杂度
- **存储空间**：O(n) - n个键值对
- **额外空间**：O(m) - m个桶的指针数组

## 🧠 内存地址分配与访问路径总结

### 完整的内存访问路径图
```
程序变量：
map (栈变量) ──→ 0x800 (LinkedHashMap结构体)
                    │
                    ├─ size: 3
                    ├─ capacity: 4  
                    └─ buckets: 0x1000 ──→ 指针数组
                                            │
                    ┌──────────────────────────┘
                    ▼
指针数组 (0x1000):   [0x2000][NULL][0x3000][0x4000]
                      │               │       │
                      ▼               ▼       ▼
链表节点:           Node1           Node2   Node3
                   (0x2000)       (0x3000) (0x4000)
                      │               │       │
                      ▼               ▼       ▼
键值对:             Pair1           Pair2   Pair3
                   (0x2100)       (0x3100) (0x4100)
                      │               │       │
                      ▼               ▼       ▼
字符串:            "hello"         "world"  "test"
                   (0x2150)       (0x3150) (0x4150)
```

### 访问路径代码示例
```c
// 访问buckets[0]第一个节点的value
char* value = map->buckets[0]->pair->value;

// 分解步骤：
// 1. map                    → 0x800 (LinkedHashMap结构体地址)
// 2. map->buckets           → 0x1000 (指针数组地址)  
// 3. map->buckets[0]        → 0x2000 (第一个Node地址)
// 4. map->buckets[0]->pair  → 0x2100 (Pair结构体地址)
// 5. map->buckets[0]->pair->value → 0x2150 (字符串地址)
```

### 内存分配的层次结构
```
Level 1: LinkedHashMap结构体     ← malloc(sizeof(LinkedHashMap))
Level 2: 指针数组buckets        ← malloc(sizeof(Node*) * capacity)
Level 3: 链表节点Node           ← malloc(sizeof(Node))
Level 4: 键值对Pair             ← malloc(sizeof(Pair))
Level 5: 字符串value            ← malloc(strlen(value) + 1)
```

### 关键理解要点
1. **`buckets[i]`的本质**：既是数组元素，也是链表头指针
2. **地址传递**：每层都是通过指针连接，形成完整的访问路径
3. **内存独立性**：每层的malloc都是独立的，需要独立释放
4. **链表遍历**：只能通过next指针，无法直接索引访问
